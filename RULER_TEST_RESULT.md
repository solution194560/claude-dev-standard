## 3) 시나리오 2 상세 분석 — `.codex/config.toml` 산출물 확인 (지시 사항 반영)

로그의 dry-run·apply 양쪽 모두에서 codex 대상 output paths 로 `AGENTS.md` 외에 `.codex/config.toml` 이 함께 출력된다.
```
[ruler:verbose] Agent OpenAI Codex CLI output paths: .../AGENTS.md, .../.codex/config.toml
```
`.gitignore` Ruler 관리 블록에도 `.codex/config.toml`·`.codex/config.toml.bak` 경로가 포함된다(6개 경로 = 3개 파일 × 2(본·백업)).

**복제본 실물 확인(apply 완료 상태, 본 시험 최종 상태)**:
```
$ ls -la /var/folders/.../ruler-test-1783819936/.codex
ls: .../ruler-test-1783819936/.codex: No such file or directory
$ cat /var/folders/.../ruler-test-1783819936/.codex/config.toml
cat: .../ruler-test-1783819936/.codex/config.toml: No such file or directory
```
**결론**: `.codex/config.toml` 은 **실제로 생성되지 않았다**. Ruler 의 codex 어댑터는 codex 대상의 잠재적 출력(AGENTS.md=지침, `.codex/config.toml`=MCP 서버 설정)을 모두 "output paths"로 선언하고 `.gitignore` 관리 블록에도 선제 등록하지만, 본 테스트는 `[mcp] enabled = false` + `--no-mcp` 로 MCP 변환을 비활성화했으므로 MCP 전용 파일인 `.codex/config.toml` 자체는 쓰지 않는다. 이는 **ruler 의 codex 어댑터 정상 동작 범위**로 판단한다 — MCP 를 켜지 않는 한 실제 파일시스템 변화는 발생하지 않고, `.gitignore` 항목은 향후 MCP 를 켤 경우를 대비한 선제 등록일 뿐이다.

PLAN §5.3 시나리오 2 통과 기준("생성물 2개 + `.gitignore` Ruler 블록 변경만 허용, 타 대상 파일 없음")과의 정합성: "타 대상 파일"은 `.cursor/` 등 **다른 AI 도구의 산출물**을 가리키며, `.codex/config.toml` 은 codex 자신의 잠재 산출물이지 타 대상 파일이 아니다. 그리고 실제 파일이 생성되지 않았으므로 "생성물 2개" 원칙도 위반되지 않는다. `git status --porcelain` 이 ` M .gitignore` 하나만 보고한 것도, 두 생성물이 이제 `.gitignore` 로 무시되기 때문이지 `.codex/config.toml` 이 추적 대상으로 나타난 게 아니다.

**판정**: 시나리오 2 **PASS**. 단, `.gitignore` 블록에 미생성 파일의 경로가 선제 등록되는 것은 Ruler 의 정상 동작이지만 문서화되지 않으면 혼동을 부를 수 있어 아래 §6 후속 항목에 기록한다.

## 4) 시나리오 11 상세 분석 — revert 후 `.gitignore` 잔류 `M` 상태

**관찰**: revert 완료(ruler 자체 보고 "cleaned: yes", 생성물 2개 제거 exit=1 로 확인) 직후 `git status --porcelain` 이 여전히 ` M .gitignore` 를 보고했다(로그 361~365행). 그러나 로그에는 이 시점의 `git diff -- .gitignore` 원문이 없어 **정확히 어떤 내용이 잔류하는지는 로그만으로 특정할 수 없다** — 이는 원시 증거 공백이다.

**정황 분석(간접 증거, 확정 아님)**:
- REPO_ROOT 의 커밋된 원본 `.gitignore` 는 정확히 `.DS_Store\nThumbs.db\n` 20바이트다(`od -c` 확인).
- apply 가 삽입한 블록의 `git diff` 는 원본 2줄 뒤에 **빈 줄 2개 + `# START ... # END` 8줄**을 추가하는 구조다(총 +10줄).
- ruler verbose 로그는 "Removed ruler block from .gitignore"라고만 보고하며, 이 문구가 "START~END" 사이만 제거하는지 삽입 시 함께 넣은 빈 줄 2개까지 제거하는지는 로그에서 확인되지 않는다.
- 만약 빈 줄 2개가 제거되지 않고 남았다면 결과 파일은 `.DS_Store\nThumbs.db\n\n\n` 형태가 되어 원본과 바이트 단위로 달라지고, 이는 `git status` 상 ` M` 으로 나타나는 것과 정합적이다.

**판정에 대한 영향**: PLAN §5.3 시나리오 11 통과 기준("생성물 2개 제거 + `.gitignore` Ruler 블록까지 원복")의 핵심은 "Ruler 블록"(추적 대상 경로 목록) 제거 여부이며, 이는 ruler 자체 보고와 이후 재적용(시나리오 12)이 정상적으로 블록을 다시 만든 사실로 뒷받침된다. 실제 파일 경로 목록이 남아있다는 증거는 없다(로그·재적용 결과 모두 "정상 블록 재구성"만 보여준다). 따라서 **핵심 기능(생성물 제거·블록 텍스트 제거)은 통과로 판정**하되, **바이트 단위 완전 원복 여부는 로그 증거 부족으로 미확정**임을 명시한다.

**판정**: 시나리오 11 **PASS (관찰 사항 첨부)** — 잔여 공백 줄 가능성이 있는 트레일링 서식 차이로, 안전·기능상 결함으로 볼 근거는 없다. 후속 회차에서 revert 직후 `git diff -- .gitignore` 를 로그에 추가하면 확정 가능(§6 후속 항목).
